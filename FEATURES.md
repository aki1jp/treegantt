# TreeGantt 機能ロードマップ — リソースビュー正式化

> 本書は **構想・ロードマップ** を記述する。確定した現行仕様は `docs/treegantt_design.md`（完全仕様書）を正とする。
> 本書の内容を実装に移すときは、設計書の該当章を更新してから着手する（CLAUDE.md の順序厳守ルール）。

## 背景 — なぜリソースビューを作り直すか

現状の `ResourceView` は「担当者×日付の**タスク件数**ヒートマップ」にすぎず、リソースビューが本来答えるべき問い—
**「各メンバーは、その人のキャパシティに対して過負荷か余裕か。いつそうなるか」**—に答えていない。

具体的な不足:

1. **件数ベースで稼働量を表現できない** — 1 日タスクも 30 日タスクも「その日 1 件」として同じ重み。工数の概念がない。
2. **実装のドリフト** — `ResourceView.tsx` は自前のインライン集計を行い、より正しい `utils/workloadCalc.ts:calcWorkloadMatrix`（`done` 除外・`endDate` 必須・範囲クランプ）は**使われていない**。
3. **キャパシティ非対応** — 1 人 1 日の処理上限という概念がなく、過負荷の「基準」がない。
4. **稼働カレンダー無視** — 土日にも稼働が加算される。
5. **ズーム非整合** — week/month でも日セル単位で、期間集計をしない。
6. **ドリルダウン無し** — セルから寄与タスクへ辿れない。

## あるべき姿（設計の正）

リソースビューの中核は次の 3 要素:

- **キャパシティ（供給）** — 各リソースの単位期間あたり処理能力。既定 1.0 FTE = 8:00/稼働日。土日・祝日 = 0。
- **デマンド（需要）** — 各タスクが担当者のキャパを期間にわたり消費する量。予定工数を稼働日へ配分して日次需要を得る。
- **稼働率 = 需要 / キャパ** — リソース×期間ごと。**過負荷（>100%）の検出と強調が主目的**。

付随要素: 稼働カレンダー考慮、ズームに整合した期間集計、サマリ指標（合計工数・ピーク稼働率）、セルから寄与タスクへのドリルダウン。

## 段階計画

### Step 1 — 「同時稼働」モデル化＋ドリフト解消（データ変更なし）

予定工数フィールドが無くても今すぐ正せる範囲。リソースビューを「同時進行中タスク数（稼働カレンダー考慮）」として一貫させる。

- 共有 `calcWorkloadMatrix` を実際に使う（`done` 除外・`endDate` 必須・範囲クランプを一本化）。
- 土日に稼働を加算しない（土日はキャパ 0）。
- ズーム整合: week/month では日次需要を**期間内ピーク（最大同時数）**で集計。
- ラベルを「同時進行タスク数」と明示、色凡例をパネル内に表示。
- セル tooltip に寄与タスク名を列挙。

### Step 2 — 工数ベースの稼働率モデル（正式仕様）

| 項目 | 決定 |
|---|---|
| 予定工数 | `Task.estimateMinutes`（整数=分）。表示ラベル「予定工数」。入力は単位トークン＋時:分（`3d` `4h` `30m` `1w` `1d 4h` `7:45`）。DB/計算は分。 |
| 入力↔保存 | 入力/表示=人間向け（単位トークン or `HH:MM`）、DB・計算=整数の分。境界で「トークン⇄分」パーサを通す（単一の真実=分）。 |
| 日換算の基準 | `1d = キャパ分/日`、`1w = 稼働日数/週 × キャパ分/日`。入力時点のキャパで分に固定（後でキャパを変えても既存値は再換算しない＝見積もり凍結。MS Project / Jira と同挙動）。 |
| 入力ヘルプ | 予定工数欄にプレースホルダ（`例: 1d 4h, 7:45, 30m`）＋ラベル横 `?` アイコンのツールチップで全書式の早見表。 |
| 日次配分 | 予定工数 ÷ タスクの稼働日数で均等配分（リーフタスクのみ。null は非計上）。 |
| ズーム集計 | 期間内ピーク（最大稼働率）。平均は使わない。 |
| 過負荷バンド | 〜80% 余裕 / 80–100% 適正 / 100–120% 注意 / >120% 過負荷。 |
| キャパシティ | 分精度（`7:45` 等）。**2段継承**: アプリ既定 → プロジェクト個別上書き（`null`=継承）。 |
| 稼働日 | 既定 月〜金。**2段継承**（アプリ既定 → プロジェクト上書き）。特定日付の祝日カレンダーは将来。 |
| 設定の保存先 | すべて DB。アプリ既定=`app_settings`（key-value）、プロジェクト上書き=`projects` の nullable 列。localStorage は個人/端末ローカルのため不採用。 |
| 継承（カスケード） | 実効値 = `プロジェクト個別値 ?? アプリ既定 ?? ハードコード既定`。 |
| 複数担当者 | `assignee` は単一据え置き。複数担当が必要なときは**子タスクへ分割**して表現する。 |

### Step 3 — 本格 PM 資源管理（今回スコープ外）

将来検討。各項目と、**今回除外する理由**は以下のとおり。

| 項目 | 内容 | 今回除外する理由 |
|---|---|---|
| 複数担当者／割合配分 | 1 タスクを複数人で分担、「50% だけ関与」等 | `assignee`（中核データ）を配列/オブジェクト化する破壊的変更。フィルタ・WBS・Import/Export 全般に波及する。当面は子タスク分割で代替できる。 |
| 名前付きリソース実体 | 担当者を文字列でなく「人」エンティティとして管理（個別カレンダー・休暇・コスト単価） | 新エンティティ＋ API＋管理画面が必要。現状の文字列担当者で当面足りる。 |
| 担当者別キャパ | 人ごとに 8:00 / 時短 6:00 等 | リソース実体化が前提。まずアプリ／プロジェクトの 2 段継承で十分。 |
| 自動レベリング | 過負荷を解消するようタスク日程を自動再配置（スムージング＝フロート内のみ／レベリング＝終了日を延ばしてでも） | スケジュールを自動で書き換える破壊的操作。プレビュー・取消・「動かさない」ロック等で UX が一気に重くなる。まず Step 2 で過負荷を「見える化」し、人手でバードラッグ調整するのが先。土台となる CPM／依存解決は既存。 |

> **中間案（将来検討）**: 自動で動かさず「過負荷の検出＋解消サジェスト」に留める軽量版。
> 例: 「6/20、A さんが 120%。①このタスクを 2 日後ろ倒し、または②B さんへ振替で解消できます」と提案し、適用は人がワンクリック。破壊度が低く、Step 2 の自然な延長線上に置ける。
