## 1. 全体構成

画面は上から「プロジェクト選択ヘッダー」「ツールバー」「ガントチャート」の 3 段で構成される。

- **プロジェクト選択ヘッダー**: ページスクロールに関係なく常に最上部に固定表示される。
- **ツールバー**: ヘッダーの直下に固定。ガントを縦スクロールしても常に見える。
- **ガントチャート**: 残余の高さ全体を使い、縦横スクロールが可能。左パネルは横スクロールしても画面左端に固定、ヘッダー行は縦スクロールしても画面上端に固定される。

## 2. プロジェクト管理

### 機能

- プロジェクトを複数作成・切り替え・削除できる。
- ヘッダーに全プロジェクト名をタブとして並べて表示する。現在表示中のプロジェクトはハイライト。
- プロジェクトを切り替えるとそのプロジェクトのタスクが即時表示される。
- プロジェクトを削除すると、そのプロジェクトに属する**全タスクも一緒に削除**される（確認ダイアログ必須）。
- プロジェクトが 0 件のとき、ガントの代わりに「プロジェクトがありません」メッセージと作成ボタンを表示する。

### 注意点・ハマりやすいところ

- プロジェクト切り替え時、リアルタイム同期（WebSocket）を切り替え後のプロジェクトに接続し直す必要がある。接続先を切り替えずに古いプロジェクトのチャンネルにつながったままになる実装ミスが起きやすい。
- 削除確認ダイアログに「タスクも全部消える」旨を明示しないとユーザーが誤操作しやすい。

## 3. タスクのデータ項目

| `parentId` | 親タスクの ID | null = ルートタスク |
| `title` | タスクタイトル | 必須。文字数上限あり |

| `priority` | 優先度 | 下表参照 |
| `progress` | 進捗率 0〜100 | 整数 |

| `endDate` | 終了日 YYYY-MM-DD | null = 未設定 |

| `createdAt` / `updatedAt` | 作成・更新日時 | 自動設定 |

## 4. ガントチャート表示

### 4.1 基本レイアウト

- **左固定列**: タスクの属性（タイトル・ステータス・優先度・進捗・担当・開始日・終了日）を表形式で表示。横スクロールしても左端に固定される。
- **右タイムライン**: 日付軸を横方向に持つ。タスクをバーで表示し、縦横スクロール可能。
- 左列の各行と右タイムラインの各行は 1:1 で対応し、高さが一致する。

> **ハマりやすい点 — 行高さのズレ**: 左パネルの各行と SVG の各行の高さを厳密に一致させないと、行数が増えるほどズレが累積する。CSS の `box-sizing: border-box` と `border-bottom` の組み合わせに注意。行の外側要素に border を持たせると「コンテンツ高さ + border 幅」が実効高さになる。高さは明示的に固定し、border は box-sizing: border-box の内側に収める。

### 4.2 マルチレベルヘッダー

ヘッダーは **年 / 月 / 週 / 日** の最大 4 段で構成される。各段は個別に表示/非表示を切り替えられる。

| 段 | ラベル例 | セル境界 |
|----|---------|---------|
| 年 | `2026` | 1 月 1 日ごと |
| 月 | `2026-05` | 月初ごと |
| 週 | `W21` | 月曜日ごと |
| 日 | `22` | 1 日ごと |

- 全段を同時に表示できる（ヘッダーが縦に高くなる）。
- デフォルトは全段表示。
- 表示段の切り替えボタンはツールバーに配置する。

> **ハマりやすい点**: ヘッダーが複数行になった場合でも sticky で上端に固定する。ヘッダー高さが動的に変わるため、高さをハードコードしてはいけない。

### 4.3 ズームレベル

「日・週・月」の 3 段階でタイムラインの縮尺を切り替えられる。縮尺が変わると 1 日あたりのピクセル幅が変わり、ヘッダーのセル幅も連動して変化する。

### 4.4 表示期間コントロール

ガントの表示範囲（開始日と期間）をツールバーで指定できる。

**自動モード（デフォルト）**
- タスクの開始日・終了日の最小〜最大から表示範囲を自動計算する。
- タスクが 0 件のときは今日を基準にした範囲を表示する。
- タスク範囲が選択期間より短くても、最低でも選択期間分を確保する（短すぎて何も見えない状態を防ぐ）。

**手動モード**
- 開始日（日付ピッカー）を指定すると、そこから選択期間分の固定範囲を表示する。
- 「今日」ボタンで今日を開始日にセットできる。
- 開始日をクリアするボタンで自動モードに戻れる。

**期間選択肢**: 2 週間 / 1 ヶ月 / 3 ヶ月 / 6 ヶ月。

> **ハマりやすい点 — 日付のタイムゾーン**: `new Date('2026-05-01')` は UTC の 0 時として解釈される。JST（UTC+9）環境だと日本時間の 5 月 1 日 9 時になる一方、`new Date()` は現地時間で返る。「今日」の日付文字列を取り出すとき `toISOString().slice(0,10)` を使うと UTC 基準になり、日本時間の深夜帯では「昨日」になることがある。日付文字列の生成は一貫してローカルタイムで行うか、UTC に統一するかを決める必要がある。

### 4.5 ガントバー

- `startDate` または `endDate` が未設定のタスクにはバーを描画しない。
- バーの幅は開始日から終了日の日数に比例する（最小でも 1 日分の幅を確保）。
- バーの色はステータスに対応した色で塗る。
- 進捗率に応じてバー内部を塗りつぶして進捗を視覚化する。
- バー内にタイトルを表示し、バーの幅を超えた部分はクリップする。
- バーをクリックするとタスク詳細モーダルが開く。

### 4.6 依存関係矢印

先行タスクが設定されているとき、ガントバー同士を矢印で結ぶ。

- 起点: 先行タスクの終了日の右端。
- 終点: 後続タスクの開始日の左端。
- 曲線（ベジェ曲線）で結ぶ。直線より見やすい。
- 起点・終点のいずれかに日付が未設定の場合は描画しない。

### 4.7 今日ライン

タイムライン上の「今日」の位置に常に縦線を表示する。非表示にする機能はない。

### 4.8 イナズマライン

実績（完了タスク）と計画（未完了タスク）の境界を示す縦線。

**X 座標の計算**:
1. `done` かつ `endDate` を持つタスクの中で最も遅い終了日 → x1
2. `done` 以外かつ `startDate` を持つタスクの中で最も早い開始日 → x2
3. イナズマ X = `(x1 + x2) / 2`

**表示条件**: `done` のタスクと `done` 以外のタスクの両方が 1 件以上あるときのみ表示する。

- ラインが今日より左 = 遅延傾向 / 右 = 進行良好の目安となる。
- ツールバーのボタンで ON/OFF を切り替えられる（デフォルト: ON）。

> **ハマりやすい点**: イナズマラインの表示条件を「完了タスクが 1 件以上あれば表示」にすると、全タスクが完了した後もラインが残り続けて混乱する。「done 以外のタスクも 1 件以上あること」を必須条件にする。

## 5. タスク一覧（左固定列）

### 5.1 列とソート

列ヘッダーをクリックするとその列でソートが切り替わる（昇順 → 降順 → 昇順 ...）。

### 5.2 インライン編集

セルをクリックすると編集モードに切り替わる。Enter またはフォーカスアウトで確定、Escape でキャンセル。

- タイトル: テキスト入力。空文字のまま確定するとキャンセル扱い。
- ステータス・優先度: セレクトボックス。選択と同時に確定。
- 進捗: 数値入力（0〜100）。範囲外の値は自動でクランプ。
- 担当者: テキスト入力。
- 開始日・終了日: 日付ピッカー。

> **ハマりやすい点 — フォーカスアウトと競合**: `onBlur`（フォーカスアウト）で保存処理を走らせると、セレクトボックスの項目をクリックしたときに `onBlur` が先に発火して意図しない保存が起きることがある。セレクトボックスは `onChange` で即確定する方式にすると安定する。

### 5.3 競合解決ダイアログ

同一フィールドを異なるブラウザで同時編集した場合、後から確定しようとした側に競合ダイアログを表示する。

- 「相手の値」と「自分の値」を並べて表示する。
- 「相手の変更を使う」: 自分の入力を破棄。
- 「自分の変更を使う」: 自分の値を強制上書き。

> **ハマりやすい点**: 競合検知は「編集を開始した時点の値」と「確定時点の現在値」を比較することで実現する。編集開始時点の値を状態として保持しておく必要がある。これを保持し忘れると競合を検知できない。

### 5.4 右クリックコンテキストメニュー

任意の行を右クリックするとメニューを表示する。

- 「編集（詳細）」: タスク詳細モーダルを開く。
- 「削除」: 確認ダイアログを表示して削除。

### 5.5 ツリー表示・折りたたみ

- `parentId` を使ってタスクをツリー構造に組み立て、それをフラットなリストとして表示する。
- 深さに応じてタイトル列を左インデントして視覚的に区別する。
- 子タスクを持つタスクの行には折りたたみボタン（▶ / ▼）を表示する。
- 折りたたむと、そのタスクの配下のタスクが一覧から非表示になる。
- 親タスクを削除したとき、子タスクの `parentId` は null（ルートへ昇格）になる。

> **ハマりやすい点 — 循環参照**: `parentId` の設定を誰でも自由に変更できると、タスク A の親をタスク B にして、タスク B の親をタスク A にするといった循環参照が起きてツリー構築が無限ループになる。循環参照チェックはアプリケーション層で必ず行う。

> **ハマりやすい点 — ソートとツリーの組み合わせ**: ソートはツリー構築の前に適用し、ツリー構造内での親子の相対順序を保つ。ソート後にフラット化することで、意図しない入れ子の順序変更を防ぐ。

### 5.6 親タスクの進捗自動計算

- 子タスクを 1 件以上持つタスクは「親タスク」とみなす。
- 親タスクの進捗 = 子タスクの進捗率の算術平均（再帰計算）。子タスクもさらに親なら同様に計算する。
- 親タスクの進捗セルはクリックしても編集モードにならない（読み取り専用）。
- 見た目で区別できるよう、バーの色や文字色を変える。

> **ハマりやすい点**: この計算値を DB に保存しようとすると、子タスクが更新されるたびに親 → 祖父母 → … と連鎖更新が必要になり複雑化する。フロントエンドの表示時のみ計算し、DB には保存しないのが正解。

### 5.7 クイック追加行（末尾固定）

タスク一覧の最終行として「＋ タスクを追加…」プレースホルダーを常時表示する。

- その行をクリックするとタイトル入力欄が現れる。
- Enter でタスクを作成、Escape またはフォーカスアウト（空文字）でキャンセル。
- 作成されるタスクはタイトル以外デフォルト値（ステータス: TODO、優先度: 中など）。

## 6. タスク詳細モーダル

「+ タスク追加」ボタン、またはガントバーのクリック、右クリック「編集（詳細）」から開く。

### 入力フィールド一覧

| フィールド | UI | 備考 |
|-----------|-----|------|
| タイトル | テキスト入力 | 必須 |
| サマリ | テキスト入力 | 任意 |
| 説明 | テキストエリア | Markdown 可、任意 |
| ステータス | セレクトボックス | TODO / Doing / DONE / 待機 |
| 優先度 | セレクトボックス | 最高 / 高 / 中 / 低 |
| 進捗率 | スライダー（0〜100） | 現在値をラベル表示 |
| 担当者 | テキスト入力 | 自由記述 |
| 開始日 / 終了日 | 日付ピッカー | 任意 |
| 親タスク | セレクトボックス | 全タスク一覧（自分自身を除く）。「なし」でルートタスク |
| 先行タスク | チェックボックスリスト | 全タスク（自分自身を除く）。複数選択可 |

### 動作

- 新規作成: 保存ボタンでタスクを作成。
- 編集: 保存ボタンで更新（変更フィールドのみ送信）。
- モーダル外クリックまたはキャンセルボタンで閉じる（変更は破棄）。

## 7. フィルタ・ソート

ツールバーに常時表示。フィルタとソートはすべてフロントエンドのメモリ上で行う（追加の API 呼び出しは不要）。

### ステータスフィルタ

| 選択肢 | 動作 |
|--------|------|
| すべて | フィルタなし |
| TODO / Doing / DONE / 待機 | 該当ステータスのみ表示 |
| **DONE 以外** | DONE 以外の全ステータスを表示（進行中のタスクを一覧したいときに便利） |

### 優先度フィルタ

最高 / 高 / 中 / 低 の単一選択。

### 担当者フィルタ

テキスト入力による**部分一致**フィルタ。

### ソート

各列ヘッダーをクリックで切り替え。再クリックで昇順/降順を反転。

| ソート対象 | ロジック |
|-----------|--------|
| 表示順（デフォルト） | `order` フィールドの数値順 |
| タイトル / 担当者 | ロケール順 |
| ステータス | todo → wip → done → wait の固定順 |
| 優先度 | critical → high → medium → low の固定順 |
| 進捗 | 数値順 |
| 開始日 / 終了日 | 日付順。未設定は末尾 |

## 8. Import / Export

ツールバーのボタンから操作する。

### JSON エクスポート

- 現在のプロジェクトの全タスクを JSON ファイルとしてダウンロードする。
- ファイルにはプロジェクト情報・エクスポート日時・全タスクデータを含める。

### CSV エクスポート

- 全タスクを CSV ファイルとしてダウンロードする。
- `predecessors`（先行タスク ID）はセミコロン区切りの文字列として 1 列に収める。

> **ハマりやすい点**: `predecessors` をカンマ区切りにすると、CSV のカラム区切りと混在してエスケープ処理が必要になる。セミコロン区切りにすることで問題を回避できる。

### インポート（JSON / CSV）

- ファイルを選択するとパースして `/projects/:id/import` エンドポイントへ送信する。
- 同一 `id` のタスクが存在する場合は**上書き（upsert）**、存在しない場合は新規作成。
- インポート完了後、最新データを再フェッチして表示を更新する。

> **ハマりやすい点**: インポート後に再フェッチしないと、画面の表示が古いままになる。Y.js のリアルタイム同期とは別に、インポート直後は明示的に最新データを取得する処理が必要。

## 9. リアルタイム同期

複数ブラウザ（タブ・別 PC）間の変更をリアルタイムに同期する。CRDT（競合のないデータ型）を使い、複数人が同時に異なるフィールドを編集しても互いの変更が失われない。

### 基本フロー（タスク更新時）

1. 自分のブラウザ: ローカルの CRDT ドキュメントを即時更新 → 自分の UI に即反映。
2. 同時に REST API（PATCH）を呼ぶ → サーバーが DB を更新 → WebSocket サーバーが全接続クライアントへブロードキャスト。
3. 他のブラウザ: WebSocket 経由でデータを受信 → UI に反映。

> **ハマりやすい点 — CRDT だけに書けば済む問題ではない**: CRDT への書き込みだけでも `onStoreDocument`（CRDT → DB への永続化コールバック）経由で DB には保存されるが、他クライアントへのリアルタイムブロードキャストが不安定になることがある。REST API も並行して呼ぶことでサーバー側から確実にブロードキャストを保証する。

### プロジェクト切り替え時の初期化

1. REST API でタスクを即時取得 → 表示（CRDT の同期完了を待たない）。
2. CRDT ドキュメントに不足しているタスクを追加（リアルタイム同期の準備）。

> **ハマりやすい点**: CRDT の同期完了（`synced` イベント）を待ってから表示しようとすると、ネットワーク遅延やオフライン時にいつまでも表示されない。REST API のレスポンスを即座に表示し、CRDT の同期は非同期で行う設計にする。

> **ハマりやすい点 — ダミープロジェクト ID**: プロジェクト未選択の状態でも CRDT の接続先を「空文字」や「_none」などのダミー ID にしていると、そのダミードキュメントが `synced=true` になるタイミングで実際のプロジェクトのデータが消える競合が起きることがある。未選択時は CRDT 接続を開始しないか、プロジェクト ID が確定してから接続する。

> **ハマりやすい点 — フレームワークの開発モード**: React などのフレームワークは開発環境でコンポーネントを 2 回マウントする場合がある（StrictMode）。WebSocket 接続を `useEffect` 内で毎回作成・破棄すると接続が失われる。接続インスタンスをコンポーネント外のキャッシュに保持し、クリーンアップで破棄しない設計にすることで回避できる。

> **ハマりやすい点 — 認証フックの副作用**: CRDT サーバー（Hocuspocus）で認証フックを定義すると、クライアント側に認証メッセージの送信を要求するプロトコルが有効になる。認証不要で使う場合は認証フックを定義しないことが正解（定義して常に `true` を返す実装にしてはいけない）。

### CRDT の構造（フィールド単位のマージ）

CRDT ドキュメントの構造は「タスク ID → フィールド辞書」の 2 層ネストにする。

```
document["tasks"]
  ├── "task-uuid-1" → { title: "...", status: "todo", ... }
  ├── "task-uuid-2" → { ... }
  └── ...
```

> **ハマりやすい点**: タスクをプレーンオブジェクト（`{ title, status, ... }` 全体）として 1 つのキーに格納する設計にすると、2 人が同じタスクの**異なるフィールド**を同時編集したときに後発の変更がオブジェクト全体を上書きして先発の変更が消える。フィールドごとに独立した CRDT エントリにすることで、フィールド単位のマージが行われる。

## 10. 接続状態バッジ

WebSocket の接続状態をツールバーに常時表示する。

| 状態 | 表示内容 |
|------|---------|
| 接続中（connected） | 緑のバッジ＋WiFi シグナルアイコン＋「接続中」 |
| 接続試行中（connecting） | 橙のバッジ＋スピナーアニメーション＋「接続中...」 |
| 未接続（disconnected） | 赤のバッジ＋WiFi ✕ アイコン＋「未接続」 |

- スピナーは CSS アニメーションで回転する。
- 状態によってアイコン・色・ラベルの 3 つすべてを変える（色だけでは色覚特性のあるユーザーに区別しにくい）。
