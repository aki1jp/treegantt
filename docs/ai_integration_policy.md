# TreeGantt AI連携方針（MCPサーバー導入の設計判断）

| 項目 | 内容 |
|------|------|

| 目的 | AI（LLMエージェント）からTreeGanttを安全に参照・将来的には操作できるようにする際の、アーキテクチャ方針を確定し経緯を残す |

## 1. 背景・経緯

TreeGanttの基本機能（タスク管理・ガント描画・リアルタイム同期・リソースビュー等）が一巡し、次の課題として「入力の手間」が挙がった。当初の要望は次の2つだった。

1. AIがAPIを使ってガントチャートを直接変更していく（自然言語での編集依頼→AIがタスクを作成・編集する）
2. AIによる仕事上のアドバイス、タスク忘れ防止の提示

検討を進める中で、要望者から「AIをTreeGantt内部に組み込みたくない。基本は人間が実施し、AIは別で参照する方向にしたい」という**方針転換**が明示された。この転換の理由（会話から読み取れる意図）は次の通り。

- TreeGantt本体のアーキテクチャ的健全性を保ちたい（AIロジック・LLM依存パッケージ・プロンプト管理などを本体に持ち込むと、本体の複雑度・依存関係・テスト範囲が増える）
- 実際にタスクを進めるのは人間である、という中心的な価値を崩したくない
- AI関連の技術（プロトコル・モデル・ベンダー）は変化が速いため、本体と疎結合にしておくことで将来の変更コストを下げたい

## 2. 中心原則

**TreeGantt本体（`api`/`frontend`）はAIの存在を一切知らない。** AIは、人間がブラウザでTreeGanttを使うのと同じ立場の「もう一つの外部クライアント」として、既存のREST API（`/api/v1/*`）を叩くだけである。

この原則から導かれる設計上の制約:

- `api`/`frontend` のソースコードには、AI関連の変更を一切加えない（依存パッケージも追加しない）。
- AIとの接続点は新設する独立パッケージ `mcp/` に閉じ込める。`mcp/` は `frontend` と同じ立場の**HTTPクライアント**であり、`api` 内部の `services/*` を直接importしない（プロセス境界をまたいだ疎結合を維持する）。
- `mcp/` が消えても、TreeGantt本体の挙動・テスト・デプロイには一切影響しない（逆も然り）。

## 3. 採用したインターフェース: MCP (Model Context Protocol)

Claude Desktop / Claude Code をはじめとするAIクライアントが標準対応する MCP を採用する。理由:

- 特定のAIベンダーにロックインしない（MCP対応クライアントであれば何でも接続できる）。
- 既存のAIクライアント側に「ツール呼び出し前の確認」「会話UI」などがすでに備わっており、TreeGantt側でチャットUIや承認フローを作り込む必要がない。

### 3.1 検討した代替案と却下理由

| 案 | 内容 | 却下理由 |
|----|------|---------|
| 汎用 OpenAPI→MCP 自動変換ブリッジ | `api` に OpenAPI 仕様を追加し、汎用ツールで REST API をそのまま MCP ツール化する | TreeGantt はタスクの親子ツリー・マイルストーン・predecessors・親の派生値（DBに保存されない）など、ドメイン固有の暗黙ルールが多い。汎用ブリッジはエンドポイントをほぼそのままツール化するため、こうしたルールをツールの説明文や引数バリデーションに落とし込めず、AIが誤った呼び出し（例: 親タスクの日付を直接更新しようとする）をしやすくなる。手書きツールなら、ツール定義そのものにドメイン知識を埋め込める。 |
| TreeGantt内部にAIロジックを実装（チャットUI・LLM呼び出しを api/frontend に追加） | 当初案。api側にLLM呼び出しエンドポイント、frontend側にチャットパネルを追加 | 要望者の方針転換により却下。本体にAI/LLM依存を持ち込まない方針に反する。 |
| AIによるタスク直接編集（書き込みツール） | `create_task`/`update_task`/`delete_task` 等をv1から提供 | 「基本人間が実施し、AIは別で参照する」という方針、および人間の承認ゲートをどこに置くかが未整理であるため、v1では見送り（詳細は4.2節）。 |
| `api` に OpenAPI 仕様（Swagger）を追加 | `@fastify/swagger` 等で既存の Fastify JSON schema から OpenAPI 仕様を自動生成する | 汎用OpenAPI→MCP自動変換ブリッジ案の前提として検討していたが、本文書で採用した手書きMCPツールは `api` のJSON schemaを自動変換しないため、AI連携の目的では不要になった。AIと無関係な一般的API文書化としては単独でやる価値があり得るが、本方針（AI連携）のスコープ外であり、必要になれば別タスクとして検討する。 |

## 4. v1スコープの決定

### 4.1 読み取り専用ツール

読み取り専用ツールは次の5つ。いずれも既存REST APIへの単純なGETマッピングであり、TreeGantt側のデータを変更しない（書き込みツールは4.2節）。

| ツール名 | 対応エンドポイント | 用途 |
|---------|-------------------|------|
| `list_projects` | `GET /api/v1/projects` | プロジェクト一覧取得 |
| `list_tasks` | `GET /api/v1/projects/:id/tasks`（status/assignee/priorityフィルタ対応） | タスク一覧取得 |
| `get_task` | `GET /api/v1/tasks/:id` | タスク単体の詳細取得 |
| `export_project` | `GET /api/v1/projects/:id/export/json` | 依存関係・担当者・進捗を含む全件コンテキストを1回で取得（アドバイス・分析用） |
| `get_settings` | `GET /api/v1/settings` | 稼働キャパシティ・稼働日（負荷分析用） |

### 4.2 書き込みツール（段階導入）

v1（4.1節）は読み取り専用で見送っていたが、要望者から「入力の手間を減らしたい」という当初の要望に立ち返り、書き込みツールを**2段階**で追加することにした。

| 段階 | ツール | 対応エンドポイント | 状態 |
|------|--------|-------------------|------|
| 段階1 | `create_task` | `POST /api/v1/projects/:id/tasks` | 実装 |
| 段階1 | `update_task` | `PATCH /api/v1/tasks/:id` | 実装 |
| 段階1 | `delete_task` | `DELETE /api/v1/tasks/:id`（`mode=subtree\|single`） | 実装 |
| 段階2 | `reorder_tasks` | `PATCH /api/v1/projects/:id/tasks/reorder` | 未着手（次回） |

一括作成（`POST /projects/:id/tasks/batch`）は段階1・2のいずれにも含めない。単発の作成・編集・削除で当初の要望（入力の手間軽減）は満たせるため、スコープを広げすぎない。必要になれば別途検討する。

**承認ゲートの方針（変更なし）:**
- **TreeGantt側に確認ダイアログやプレビューUIを新設しない**。Claude Code / Claude Desktop などのMCPクライアントは、未許可のツールを実行する前にユーザーへ確認を求める仕組みを標準で持つため、その確認プロンプトを「人間の承認ゲート」として利用する。
- MCPサーバー側に追加ロジック（差分プレビューの生成等）を持たせず、既存REST APIをそのまま呼ぶだけの薄いラッパーに留める（3章の原則を維持する）。
- ツールの入力スキーマ（zod）に `status`/`priority` の enum・`progress` の 0–100 範囲・`title` の文字数上限など、`api` 側のバリデーションと同じドメインルールを埋め込む（3.1節で「手書きツールの利点」として挙げた点を実践する）。
- `titleColor`/`titleBgColor`（見た目の色）と `order`（ドラッグ順）は段階1・2のツール入力に含めない。AIによる自然言語編集の対象は内容（タイトル・日付・状態・担当者等）であり、装飾・手動並び順はスコープ外。

**変更元の記録（`source: human | ai:<name>`）は今回も追加しない。** 4.2節（旧稿）で「書き込みツールを追加する時点で再検討する」としていたが、まさにそのタイミングである今、改めて検討した結果、引き続き見送る。理由: 承認ゲートをAIクライアント側の確認プロンプトに委ねているため、少なくとも段階1の時点ではTreeGantt側で「これはAIが書いた」と表示できなくても運用上の実害が小さい。`api`/DBスキーマの変更を伴う（スコープが本パッケージ内で完結しなくなる）ため、実際に監査上の必要性が出てから着手する。

### 4.3 認証

v1では認証を導入しない。MCPサーバーはローカルの `http://localhost:4000`（`API_BASE_URL` 環境変数で変更可）を直接叩く。

理由: 現状のTreeGantt APIはブラウザからのアクセスも無認証（ゲスト単一ユーザーモデル）であり、同一マシン上で動くMCPサーバーに認証を課しても脅威モデルは変わらない（防御力が実質増えない）。

**将来、次のいずれかが発生した時点でAPIキー等の認証導入を検討する:**

1. TreeGantt APIをlocalhost外に公開する（例: チームで1台のTreeGanttサーバーを共有する）
2. 「AIが行った変更」と「人間が行った変更」を監査ログ上で区別したくなる

> 補足（段階1導入時点）: 条件2は「書き込み系ツールを追加すること」自体ではなく、追加した**結果として区別したくなること**を指す。段階1（4.2節）で書き込みツールを実際に追加したが、区別の必要性自体はまだ生じていないため、認証もsource記録も導入しない判断を維持する。

### 4.4 AIの助言・タスク忘れ防止の表示場所

AIの助言（進捗遅延リスク・担当者の詰め込みすぎ・放置されたタスクの指摘等）は、**AIクライアントとのチャット会話内で完結させる**。TreeGantt側（frontend）に通知パネルや助言保存用のAPIを新設しない。

理由:
- TreeGantt側のコード変更なしに、MCPツール（4.1のexport_project等）を使ったAIとの対話だけで価値が出せる。
- 助言をTreeGantt内のデータとして永続化すると、「本体はAIの存在を知らない」という中心原則（2章）に反する。
- 将来、助言をTreeGantt UI上にも表示したくなった場合は、別途スコープとして再検討する（本文書のスコープ外）。

## 5. パッケージ構成

新規パッケージ `mcp/` を `api`/`frontend` と並ぶ独立パッケージとして追加する。

- トランスポート: **stdio**（Claude Code/Desktopがコマンドとして起動する形。ネットワーク越しの常時稼働サーバーにはしない）
- 依存: `@modelcontextprotocol/sdk`
- `api`/`frontend` のコードは一切変更しない

詳細なファイル構成・ツール実装は `docs/treegantt_design.md` §18 を参照。

## 6. まとめ

| 論点 | 決定 |
|------|------|
| AIをTreeGantt本体に組み込むか | 組み込まない。外部MCPクライアントとして扱う |
| MCPサーバーの置き場所 | TreeGanttリポジトリ内 `mcp/`（TreeGantt専用ドメイン知識を持つため） |
| MCPサーバーの実装方式 | 既存REST APIをHTTP経由で叩くだけの薄いクライアント（api内部を直接importしない） |
| 読み取りツール範囲 | 5ツール（4.1節） |
| 書き込みツール | 段階1（`create_task`/`update_task`/`delete_task`）導入。段階2（`reorder_tasks`）は未着手。承認ゲートはMCPクライアント側の確認プロンプトに委ねる |
| 認証・source記録 | 書き込みツール導入後も引き続き見送り。監査ログでの区別が実際に必要になったら再検討 |
| 助言の表示 | AIクライアントとのチャット内で完結。TreeGantt側にUI/APIを追加しない |
